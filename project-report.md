# Отчёт по проекту `RH AI memory and EdAgent`

## 1. Общая информация о проекте

### 1.1. Название проекта

`RH AI memory and EdAgent`

### 1.2. Краткое описание

Проект представляет собой standalone-платформу для анализа рынка вакансий, поиска индустриальных партнёров, подготовки коммуникаций с компаниями и формирования проектных заданий для образовательной программы. Система объединяет аналитический контур, операторский интерфейс, RAG-механизм, слой памяти взаимодействий и ML-модуль адаптивных рекомендаций.

Ключевая идея проекта состоит в том, чтобы автоматизировать путь от анализа внешнего рынка до подготовки конкретных проектных инициатив с компаниями. Платформа помогает:

- анализировать вакансии и извлекать востребованные компетенции;
- определять разрыв между учебной программой и требованиями рынка;
- находить и ранжировать компании для сотрудничества;
- генерировать коммуникации и сопровождать outreach-процесс;
- фиксировать ответы, соглашения и проектные артефакты;
- использовать retrieval-augmented generation для более обоснованной генерации писем и brief-документов;
- накапливать память взаимодействий и адаптировать стратегию коммуникации.

### 1.3. Актуальность

В образовательных и проектно-ориентированных программах часто существует разрыв между тем, чему обучают студентов, и тем, что требуется индустрии. Кроме того, взаимодействие с потенциальными компаниями-партнёрами обычно ведётся вручную: поиск компаний, анализ потребностей, подготовка писем, сопровождение переписки и оформление договорённостей требуют значительного времени.

Проект решает сразу несколько практических задач:

- ускоряет анализ рынка труда;
- формирует более обоснованные индустриальные гипотезы;
- помогает образовательной программе выстраивать системную работу с партнёрами;
- делает процесс более воспроизводимым и управляемым;
- демонстрирует применение RAG и ML в прикладной задаче поддержки оператора.

## 2. Цель и задачи проекта

### 2.1. Цель

Разработать платформу, которая помогает образовательной программе анализировать рынок вакансий, находить релевантных индустриальных партнёров, сопровождать коммуникации с ними и формировать проектные задания, используя retrieval-augmented generation, память взаимодействий и адаптивные ML-рекомендации.

### 2.2. Основные задачи

В рамках проекта были поставлены и реализованы следующие задачи:

1. Построить инженерный фундамент в виде монорепозитория с backend, worker, web-интерфейсом, базой данных, очередями и ML-сервисом.
2. Реализовать pipeline анализа вакансий через `HH`.
3. Добавить извлечение и нормализацию компетенций.
4. Построить модуль gap analysis между программой и рынком.
5. Реализовать поиск компаний, скоринг и shortlist.
6. Реализовать генерацию коммуникаций и процесс их утверждения.
7. Реализовать outreach runtime в демонстрационном режиме.
8. Добавить обработку ответов, соглашений, project brief и каталог проектов.
9. Реализовать память агента и адаптивные рекомендации.
10. Добавить отдельный ML-сервис для удалённых рекомендаций и оценки качества.
11. Добавить RAG-контур с документами, чанками, embeddings и retrieval.
12. Построить operator console в браузере для полного e2e-сценария.

## 3. Границы проекта и принятый scope

### 3.1. Что входит в текущий scope

В финальном варианте проекта были зафиксированы следующие продуктовые границы:

- `HH` используется как основной и единственный источник вакансий;
- канал коммуникации реализован как `email`, но в демонстрационном режиме он работает через `simulated` provider;
- используется baseline `RAG`, встроенный в платформу;
- каталог проектов реализован как внутренний модуль платформы;
- используется память взаимодействий и ML-слой рекомендаций;
- приложение работает как standalone-система без обязательной зависимости от внешней LMS или CRM.

### 3.2. Что сознательно исключено из scope

Из текущего релиза были осознанно исключены:

- интеграция с `LinkedIn`;
- второй источник вакансий;
- интеграция с системой `ПроКомпетенции`;
- обязательная реализация `QLoRA`, `RFT`, `NARS`;
- обязательная интеграция с реальным почтовым провайдером для демонстрации.

Такое решение позволило сфокусироваться на основном рабочем сценарии и довести его до полноценного end-to-end состояния.

## 4. Общая архитектура системы

### 4.1. Архитектурный подход

Проект реализован как `TypeScript`-монорепозиторий с отдельным `Python` ML-сервисом. Архитектура разделена на несколько контуров:

- `apps/api` — HTTP API, бизнес-логика, авторизация, RBAC, orchestration;
- `apps/worker` — фоновые задачи, ingestion, scoring, outreach processing;
- `apps/web` — браузерный интерфейс оператора на `React`;
- `packages/database` — слой доступа к данным поверх `Prisma`;
- `packages/domain` — доменные типы;
- `packages/integrations` — внешние интеграции и адаптеры;
- `packages/scoring` — скоринговая логика;
- `packages/ai` — генерация текстов, recommendation policy, RAG helpers;
- `packages/config` — конфигурация окружения;
- `services/ml` — отдельный `FastAPI` сервис рекомендаций и evaluation.

### 4.2. Инфраструктурные компоненты

Система использует следующие технологические компоненты:

- `Node.js` + `TypeScript` для backend, worker и web runtime;
- `React` для фронтенда;
- `PostgreSQL` как основную базу данных;
- `Redis` как инфраструктурный сервис для runtime-контуров;
- `Prisma` как ORM и инструмент миграций;
- `Docker Compose` для локального развёртывания всех сервисов;
- `Python` + `FastAPI` для ML-сервиса.

### 4.3. Сервисы и порты

В локальном окружении используются следующие сервисы:

- `Web`: `http://localhost:3000`
- `API`: `http://localhost:4000`
- `API health`: `http://localhost:4000/health`
- `Worker health`: `http://localhost:4100/health`
- `PostgreSQL`: `localhost:5433`
- `Redis`: `localhost:6379`

## 5. Структура монорепозитория

Структура проекта организована следующим образом:

- `apps/api` — backend API;
- `apps/worker` — фоновые job handlers;
- `apps/web` — React-клиент и web shell;
- `packages/database` — репозитории и мапперы БД;
- `packages/domain` — доменные типы;
- `packages/integrations` — адаптеры HH, email, NLP, remote ML;
- `packages/scoring` — подсчёт scores;
- `packages/ai` — генерация summary, drafts, materials, briefs и recommendation policy;
- `packages/config` — env/config parsing;
- `packages/shared` — общие runtime helpers;
- `prisma` — схема и миграции БД;
- `services/ml` — Python ML service.

На поздних этапах разработки проект был дополнительно отрефакторен: крупные файлы были разрезаны на роуты, контроллеры, сервисы, job-модули и репозитории. Это позволило сделать архитектуру более поддерживаемой.

## 6. Модель данных

### 6.1. Основные сущности

В основе предметной области лежат следующие сущности:

- `User` — пользователь системы;
- `Industry` — индустриальное направление;
- `Competency` — компетенция;
- `ProgramCompetency` — связь компетенции с программой;
- `Vacancy` — вакансия;
- `Company` — компания-партнёр;
- `CompanyContact` — контактное лицо компании;
- `CompanyScore` — скоринговая оценка компании;
- `OutreachCampaign` — коммуникационная кампания;
- `MessageDraft` — черновик сообщения;
- `Message` — отправленное сообщение;
- `MessageEvent` — события жизненного цикла сообщения;
- `Reply` — ответ компании;
- `PartnerAgreement` — соглашение с компанией;
- `CommunicationPackage` — коммуникационные материалы;
- `ProjectBrief` — проектный brief;
- `AgentMemoryEvent` — событие памяти агента;
- `IndustrySource` — источник вакансий для индустрии;
- `IngestionRun` — запуск процесса ingestion;
- `BackgroundJob` — фоновая задача;
- `SystemSetting` — системные настройки;
- `AuditLog` — аудит действий.

### 6.2. Сущности RAG

Для реализации retrieval-augmented generation были добавлены:

- `KnowledgeDocument` — документ базы знаний;
- `KnowledgeChunk` — чанк документа с embedding-представлением.

`KnowledgeDocument` хранит:

- тип документа;
- заголовок;
- ссылку на источник;
- исходный текст;
- метаданные.

`KnowledgeChunk` хранит:

- ссылку на документ;
- индекс чанка;
- текст чанка;
- число токенов;
- embedding.

Это позволяет строить knowledge base на уровне конкретной компании и выполнять поиск по релевантным фрагментам данных.

## 7. Основной пользовательский и бизнес-флоу

Основной рабочий сценарий системы выглядит так:

`вакансии HH -> извлечение компетенций -> gap analysis -> поиск компаний -> scoring/shortlist -> генерация писем -> outreach -> ответы -> agreement -> brief -> project catalog -> memory/ML adaptation`

### 7.1. Анализ рынка вакансий

Платформа подключается к `HH` и получает вакансии по выбранному индустриальному направлению. Для каждой вакансии сохраняются:

- заголовок;
- компания;
- регион;
- тип занятости;
- опыт;
- требования;
- обязанности;
- ссылка;
- дата публикации;
- salary-данные, если они доступны.

Реализованы режимы работы `HH`:

- `live`;
- `auto`;
- `fixtures`.

В демонстрационном режиме возможен fallback, но основная архитектура ориентирована на реальный поток данных.

### 7.2. Извлечение и нормализация компетенций

После загрузки вакансий система извлекает из них компетенции. Используется normalization layer с canonical names, alias-правилами и category mapping. Дополнительно есть preview-механизм для проверки качества extraction без полного ingestion-процесса.

На этом этапе формируется структурированный список навыков и требований, отражающий индустриальные сигналы рынка.

### 7.3. Gap analysis

Далее extracted competencies сопоставляются с программными компетенциями. Результатом становится матрица разрыва между образовательной программой и индустриальными требованиями. Это позволяет увидеть:

- какие компетенции уже покрыты;
- какие компетенции покрыты частично;
- какие компетенции отсутствуют;
- какие направления стоит усилить.

### 7.4. Поиск и ранжирование компаний

На основе vacancy intelligence и дополнительных сигналов система формирует пул компаний. Для них вычисляется score по нескольким измерениям:

- competency fit;
- reputation;
- education readiness.

Дополнительно формируется explainable score breakdown, чтобы оператор видел, почему компания получила тот или иной итоговый балл.

### 7.5. Shortlist и ручная верификация

Компании попадают в shortlist и далее могут быть вручную переведены по pipeline:

- `discovered`
- `shortlisted`
- `approved`
- `contacted`
- `replied`
- `partnered`

Ручной review не затирается последующими discovery job, что важно для управляемого операционного процесса.

### 7.6. Генерация коммуникаций

Для выбранной компании система генерирует drafts сообщений. Поддерживаются различные `tone`:

- `formal`
- `neutral`
- `friendly`

Оператор может:

- создать draft;
- просмотреть его;
- утвердить или отклонить;
- сохранить версионность draft-артефактов.

### 7.7. Outreach runtime

После утверждения draft оператор запускает outreach campaign. В текущем scope используется `simulated outreach`, чтобы не зависеть от внешних email-провайдеров при демонстрации. Тем не менее платформа воспроизводит весь жизненный цикл сообщения:

- постановка в очередь;
- отправка;
- delivery status;
- reply;
- follow-up;
- escalation.

### 7.8. Replies и negotiation outcome

Ответы классифицируются по категориям:

- `interest`
- `decline`
- `question`
- `meeting`

После положительного ответа оператор может зафиксировать результат переговоров. Эти данные записываются в память системы и используются в дальнейшем для адаптации рекомендаций.

### 7.9. Agreements, brief и project catalog

После успешной коммуникации создаётся `PartnerAgreement`. На его основе генерируется `ProjectBrief`, который связывает:

- компанию;
- соглашение;
- проектную задачу;
- роли;
- компетенции.

Далее на основе brief-артефактов формируется внутренний каталог проектов.

### 7.10. Memory и ML adaptation

На всех этапах система пишет `AgentMemoryEvent`, что позволяет:

- отслеживать историю действий;
- адаптировать tone и follow-up cadence;
- анализировать успешные и неуспешные сценарии;
- использовать эту историю в recommendation и RAG-контуре.

## 8. Реализация RAG

### 8.1. Почему RAG нужен в проекте

Тема проекта связана с интеллектуальной системой, которая должна не просто хранить структурированные данные, но и использовать накопленные знания для улучшения генерации. Поэтому в проекте был добавлен baseline `RAG`.

RAG нужен для того, чтобы генерация писем и project brief строилась не только на жёстко собранных полях профиля компании, но и на наиболее релевантных фрагментах ранее накопленной информации.

### 8.2. Источники данных для knowledge base

База знаний компании формируется из следующих источников:

- профиля компании;
- вакансий `HH`;
- communication packages;
- project briefs;
- replies;
- memory events.

Таким образом, knowledge base агрегирует как внешние данные, так и внутреннюю историю взаимодействий.

### 8.3. Chunking

Каждый документ базы знаний разбивается на чанки. Это нужно для того, чтобы:

- уменьшить размер единицы retrieval;
- повысить точность поиска релевантного контекста;
- использовать только наиболее полезные фрагменты информации при генерации.

### 8.4. Embeddings

В проекте реализован локальный deterministic embedding baseline. Это не внешний SaaS и не отдельная vector DB, а встроенный механизм, который создаёт embedding-представление для каждого чанка и для пользовательского запроса.

Такой подход выбран как технически достаточный baseline для учебного проекта:

- он позволяет продемонстрировать принцип embeddings;
- не требует внешнего сервиса;
- хорошо подходит для локального demo-сценария.

### 8.5. Similarity search

После построения embedding для пользовательского запроса система вычисляет similarity между запросом и сохранёнными чанками. Используется `top-k retrieval`, то есть выбираются наиболее релевантные фрагменты базы знаний.

### 8.6. Grounding generation

Выбранные чанки используются как grounding context в генерации:

- `drafts`
- `project briefs`

Это означает, что генератор опирается не только на профиль компании и общие данные, но и на retrieved knowledge chunks, релевантные текущему запросу.

### 8.7. RAG API

Для работы с RAG добавлены отдельные ручки:

- `POST /rag/reindex`
- `GET /rag/search`
- `POST /rag/search`

Через них можно:

- переиндексировать knowledge base для конкретной компании;
- выполнить retrieval по запросу;
- получить top-k релевантных чанков с similarity-оценками.

### 8.8. Значение RAG для проекта

Добавление RAG делает проект ближе к заявленной теме и даёт следующие преимущества:

- повышает обоснованность генерации;
- делает drafts и briefs более контекстно-зависимыми;
- показывает использование retrieval-augmented generation в прикладной системе;
- связывает разрозненные артефакты проекта в единую knowledge base.

## 9. ML-сервис и адаптивные рекомендации

### 9.1. Отдельный Python ML-сервис

В проекте реализован отдельный сервис `services/ml` на `FastAPI`. Он используется для:

- remote adaptive recommendation;
- оценки качества рекомендаций;
- сравнения локальной эвристики и удалённого ML-движка.

### 9.2. Логика рекомендаций

ML-сервис анализирует статистику:

- число событий;
- число reply;
- долю positive reply;
- meeting rate;
- историю tone;
- наличие decline/question signals;
- количество отправленных follow-up.

На основе этого сервис рекомендует:

- preferred tone;
- recommended follow-up days;
- confidence;
- explanation reasons.

### 9.3. Fallback-модель

Если ML-сервис недоступен, backend и worker могут откатиться на локальную `TypeScript`-эвристику. Это повышает устойчивость системы и позволяет демонстрировать её локально без риска полной недоступности ML-контура.

### 9.4. Evaluation

В проекте реализована ручка `POST /ml/evaluations/run`, которая позволяет оценивать:

- количество проверенных сценариев;
- average confidence;
- distribution по рекомендуемым tone;
- average follow-up days;
- число случаев, где remote recommendation отличается от baseline.

Таким образом, Phase 7–8 представлены не только как архитектурная идея, но и как измеримый слой.

## 10. Память агента

### 10.1. Назначение memory layer

Память агента нужна для того, чтобы платформа:

- сохраняла историю взаимодействий;
- могла учитывать предыдущий опыт;
- меняла стратегию коммуникации на основе накопленных данных;
- использовала прошлые события как источник контекста для RAG и ML.

### 10.2. Что записывается в память

В `AgentMemoryEvent` записываются события:

- generation drafts;
- approval drafts;
- outreach delivery/failure;
- reply processing;
- escalation;
- agreements;
- project brief generation;
- communication package generation;
- negotiation outcomes.

### 10.3. Как используется память

Память применяется:

- при расчёте adaptive recommendations;
- при генерации follow-up cadence;
- при построении memory overview;
- как часть knowledge base в RAG-контуре.

## 11. Авторизация и RBAC

### 11.1. Поддерживаемые способы входа

Система поддерживает:

- локальный admin login;
- Google auth;
- signed bearer sessions.

### 11.2. Защита API

Публичными остаются только:

- `GET /health`
- `auth/*`

Все основные бизнес-ручки требуют авторизации.

### 11.3. Роли пользователей

В системе используются три роли:

- `admin`
- `manager`
- `operator`

#### `admin`

Имеет полный доступ ко всем функциям системы.

#### `manager`

Может вести основной бизнес-флоу, но не должен менять системные настройки и конфигурацию источников.

#### `operator`

Имеет доступ к просмотру и безопасным подготовительным действиям, но не может выполнять критические approve/send/system-level операции.

RBAC реализован как часть production baseline и уже используется для ограничения доступа к бизнес-операциям.

2. Выполнить вход в систему.
3. Нажать `Bootstrap индустрии`.

8. Нажать `Сгенерировать draft`.
9. Нажать `Подтвердить draft`.
10. Нажать `Отправить campaign`.
11. Нажать `Симулировать reply`.
12. Нажать `Сохранить outcome по reply`.
13. Нажать `Создать agreement`.
14. Нажать `Сгенерировать brief`.
15. Нажать `Сгенерировать материалы`.

### 13.3. Ожидаемый результат

После прохождения этого сценария система должна показать:

- вакансии, загруженные из `HH`;
- extracted competencies и gap analysis;
- найденные компании и их scores;
- созданные drafts;
- сообщения и message events;
- reply и negotiation outcome;
- agreement;
- project brief;
- communication package;
- memory events;
- ML recommendations.

### 13.4. Проверка RAG

Отдельно после появления компании, материалов, replies и brief можно проверить RAG:

1. Переиндексировать knowledge base для компании.
2. Выполнить `rag/search`.
3. Убедиться, что возвращаются top-k чанки с relevance.
4. Повторно сгенерировать draft или brief.

Ожидаемый результат:

- генерация использует не только статический профиль компании;
- в процесс вовлечён retrieved context из knowledge base.

## 15. Ограничения текущей реализации

Несмотря на значительный объём реализованной функциональности, в проекте остаются ограничения:

- интеграция с `LinkedIn` не реализована;
- второй источник вакансий отсутствует;
- интеграция с `ПроКомпетенции` отсутствует;
- delivery email работает в simulated-режиме для demo;
- используется baseline local RAG, а не полноценная внешняя vector DB;
- advanced research memory approaches (`QLoRA`, `RFT`, `NARS`) не входят в текущий релиз;
- production-level observability, compliance и автоматические test suites ещё требуют дальнейшего развития.

Однако эти ограничения не мешают демонстрации основного сценария и не нарушают целостность core-продукта.
